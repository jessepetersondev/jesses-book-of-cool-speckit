## 1. Constitution

```text
[$speckit-constitution](/home/ai/clawd/projects/kalshi-quant-dashboard/.agents/skills/speckit-constitution/SKILL.md) Kalshi Quant Dashboard is a production trading observability system. Non-negotiable rules:
1. Security first. The browser never connects directly to RabbitMQ, Kalshi, or any secret-bearing service. All secrets stay server-side. Authentication and RBAC are required.
2. Contract first. Every network and event boundary must have shared TypeScript contracts and runtime validation. No untyped or unvalidated payload may cross a boundary.
3. Live/historical parity. Every live event shown in the UI must be durably queryable from persisted history with stable identifiers, timestamps, and source metadata.
4. Event integrity. Ingestion must be idempotent, replay-safe, deduplicated by canonical identifiers, and explicit about degraded or missing upstream data.
5. Observability first. Structured logging, metrics, tracing, health checks, and alertable failure states are mandatory.
6. Extensibility. BTC, ETH, SOL, and XRP are first-class initial strategies, but future strategies must be addable without front-end rewrites.
7. Performance. Use server-side pagination, filtering, aggregation, and virtualized rendering for large datasets. Do not load unbounded history into the browser.
8. Accessibility and UX. Keyboard accessibility, screen-reader support, deterministic loading/error/empty/reconnecting states, and a dashboard-appropriate dark theme are required.
9. Testing. Unit, integration, contract, and end-to-end tests are required for all critical flows.
10. Documentation and operations. Local setup, runbooks, environment docs, schema docs, and troubleshooting docs are required before work is considered done.
```

## 2. Specify

```text
[$speckit-specify](/home/ai/clawd/projects/kalshi-quant-dashboard/.agents/skills/speckit-specify/SKILL.md) Build a new internal product called Kalshi Quant Dashboard. It is the control plane and observability dashboard for all of my Kalshi quant systems. It must unify real-time and historical visibility across the following local sibling repositories: ../kalshi-btc-quant, ../kalshi-eth-quant, ../kalshi-sol-quant, ../kalshi-xrp-quant, ../kalshi-integration-event-publisher, and ../kalshi-integration-executor.

The dashboard must let an operator or developer:
- see every live trade decision generated by each quant system
- see every trade intent, every actual trade submitted, every fill/update/result, and every skipped trade
- see why a trade was skipped, delayed, retried, rejected, or dead-lettered
- see the end-to-end lifecycle of a decision from strategy emission through publisher, RabbitMQ, executor, Kalshi submission, and terminal result
- see PnL by strategy, symbol, market, and aggregate portfolio for 24h, 7d, 30d, MTD, YTD, and all-time, plus custom range
- compare BTC, ETH, SOL, and XRP side-by-side, while allowing future strategies to appear without hard-coded rewrites
- inspect per-decision and per-trade audit trails with correlation IDs, timestamps, routing keys, queue/exchange metadata, order IDs, Kalshi IDs, retry counts, and final outcome
- monitor the centralized event publisher, RabbitMQ health, queue depth, consumer count, oldest message age, DLQ size, publish failures, unroutable events, and reconnect status
- receive visible alerts for missing heartbeats, stuck queues, growing DLQs, failed consumers, stale PnL, ingest failures, or missing terminal events

Core views required:
- Overview dashboard with global health, aggregate PnL, live decision feed, live trade feed, skip count, queue health, and recent alerts
- Strategy list and strategy detail pages for BTC, ETH, SOL, XRP and future systems
- Decisions page with filterable/searchable table and detail drawer
- Trades and order lifecycle page with filterable/searchable table and detail drawer
- Skipped trades page with skip taxonomy and root-cause breakdown
- PnL analytics page with summaries, time series, attribution, and win/loss metrics
- Pipeline and RabbitMQ operations page that visualizes the event flow and bottlenecks
- Alerts/incidents page and system health page

Data integrity requirements:
- the UI must never invent state from partial data; it must show exact raw facts and normalized derived states
- live updates must converge with persisted history
- duplicated messages must not create duplicated facts in the UI
- every decision and trade must be traceable across systems via correlation IDs
- missing or degraded upstream data must be surfaced explicitly as degraded, not silently hidden

Operational requirements:
- internal authentication and role-based access are required
- the app must support live updates, historical search, CSV export, and deep-linking to decision/trade detail states
- the app must be desktop-first but still usable on tablet widths
- all times must support UTC and local display
- error, loading, empty, reconnecting, and degraded modes must be first-class
- the system must be testable locally with seeded/sample data and with real RabbitMQ integration
- the output should be production ready, not a demo, scaffold, or mock-only dashboard

Success criteria:
- an operator can tell in under a minute what each strategy is doing, which trades executed, which were skipped, why they were skipped, how PnL is trending, and whether the publisher/RabbitMQ/executor pipeline is healthy
- a developer can trace a single decision or trade end-to-end without reading raw logs
- onboarding a new quant repo for another asset requires configuration and contract mapping, not a front-end rewrite
```

## 4. Plan

```text
[$speckit-plan](/home/ai/clawd/projects/kalshi-quant-dashboard/.agents/skills/speckit-plan/SKILL.md) Build this as a new TypeScript monorepo named kalshi-quant-dashboard with a production-ready React frontend and a server-side ingestion/read layer.

Technical choices:
- Frontend: Vite, React with Hooks, TypeScript, Redux Toolkit, RTK Query, React Router, a shared component system, accessible table/form primitives, and production-grade charting.
- Backend: TypeScript service that consumes RabbitMQ events, normalizes them, stores them durably, exposes REST APIs for historical queries and WebSocket or SSE for live updates, and polls or scrapes RabbitMQ management data for operational metrics.
- Data: PostgreSQL for normalized entities and time-series aggregates. Use durable migrations, seed scripts, and representative fixtures.
- Contracts: shared TypeScript contracts plus runtime validation schemas for every external payload and internal event. No unvalidated payload crosses a boundary.
- Testing: unit, integration, contract, and Playwright end-to-end tests. Use containerized local dependencies for RabbitMQ and Postgres.
- Delivery: Dockerfiles, docker-compose/local stack, CI pipelines for lint, typecheck, test, and build, plus environment docs and runbooks.

Architecture requirements:
- do not connect the browser directly to RabbitMQ or Kalshi
- inspect the sibling local repos named exactly ../kalshi-btc-quant, ../kalshi-eth-quant, ../kalshi-sol-quant, ../kalshi-xrp-quant, ../kalshi-integration-event-publisher, and ../kalshi-integration-executor to derive actual message schemas, routing keys, queue names, identifiers, lifecycle states, and health signals from the code
- if the sibling repos are inconsistent, define a canonical normalized dashboard event schema and build adapter/mapping layers
- model strategies as data-driven systems so future assets can be added by configuration or onboarding records, not front-end code rewrites
- include historical persistence, idempotent ingestion, deduplication, replay safety, and gap detection/reconciliation for live events
- support aggregate portfolio views and strategy drill-down views
- support alerts, heartbeats, queue health, DLQ inspection metadata, and pipeline latency tracking
- support search by decision id, order id, client order id, Kalshi order id, correlation id, routing key, market ticker, and symbol
- include role-based auth, audit logging, feature flags/configuration, structured logs, metrics, tracing, health checks, and degraded-mode UX
- generate plan.md plus data-model.md, contracts/, research.md, and quickstart.md with realistic validation scenarios

Implementation phases required in the plan:
1. Repository foundation, auth shell, layout, contracts, ingestion skeleton, database schema, local dev stack
2. Live decisions/trades feeds and overview page
3. PnL analytics and historical queries
4. Pipeline/RabbitMQ operations and alerts
5. Hardening: tests, performance, docs, deployment, accessibility, security, and reconciliation

Non-negotiable outputs from planning:
- normalized event/data model
- REST contract set and live update contract set
- page map and state management design
- ingestion and reconciliation design
- test strategy
- ops/deployment plan
- exact assumptions about upstream repos called out explicitly rather than hidden
```
